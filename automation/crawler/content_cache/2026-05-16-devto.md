# WebGPU: The Secret Weapon for Browser Video Processing

> **TL;DR:** Video processing on the CPU breaks down at scale. WebGPU changes everything — decode in the browser, hand each frame to the GPU, keep the expensive work on the GPU for real-time effects and analysis.

Video processing on the CPU is the naive path. It works for small demos and short clips. It falls apart when frame sizes go up, frame rates stay high, and every effect wants to touch every pixel.

A single 4K frame has more than 8 million pixels. At 60fps, that's nearly 500 million pixels per second before blur, color correction, or motion analysis. That is too much for JavaScript on the CPU. **Video processing is GPU work.**

## The pipeline that works

The key insight: use WebCodecs + WebGPU together.

1. **WebCodecs** decodes the video, giving you `VideoFrame` objects
2. **WebGPU** imports them with `GPUDevice.importExternalTexture({ source: videoFrame })`
3. The frame becomes an internal `rgba8unorm` texture — the stable input for all processing

**What's NOT in the pipeline:** no `getImageData()`, no `readPixels()`, no full-frame CPU readback. Reading from GPU forces CPU-GPU synchronization and moves data in the wrong direction. If you need CPU data, ask for the smallest result — not the whole frame.

## Effects as WGSL shaders

Every effect is a fragment shader:

```wgsl
@group(0) @binding(0) var samp: sampler;
@group(0) @binding(1) var videoTex: texture_2d<f32>;

@fragment
fn fs_main(in: VertexOutput) -> @location(0) vec4f {
  let color = textureSample(videoTex, samp, in.uv);
  return vec4f(color.rgb, color.a);
}
```

The registry includes: hue shift, brightness, contrast, saturation, levels, invert, pixelate, kaleidoscope, mirror, RGB split, and box blur. Each one samples the input texture, computes new pixels, writes to a render target.

## Ping-pong for effect chains

GPU passes can't safely read and write the same texture. The solution: a ping-pong pair, swapping input/output after each effect. One effect or five — the output is always a GPU texture view ready for the next stage.

## Analysis on the GPU too

Not just visual filters. The project computes:
- **Histogram** — pixel intensity distribution
- **Waveform** — broadcast-standard visualization
- **Vectorscope** — color analysis
- **Optical flow** — motion estimation via compute shaders

The optical flow pipeline runs grayscale conversion, pyramid downsampling, spatial/temporal gradients, and Lucas-Kanade flow — all on GPU. CPU only gets compact stats: motion totals, scene-cut hints.

## The takeaway

The split between "JavaScript edits video" and "JavaScript orchestrates a GPU video pipeline" is the difference between something that works for demos and something that scales. WebGPU makes the second one possible in the browser.
