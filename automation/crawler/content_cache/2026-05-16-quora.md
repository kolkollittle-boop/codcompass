What makes WebGPU particularly powerful for video processing in the browser is that it fundamentally changes the data flow architecture.

Traditional browser video processing follows a CPU-first pattern: decode → pull frame to JavaScript memory → process pixels in JS → render. This works for toy demos but breaks down immediately at scale. A single 4K frame at 60fps means ~500 million pixels per second to process — way too much for JavaScript.

WebGPU changes this by keeping the hot path entirely on the GPU:

**The key technique is `importExternalTexture`.** Instead of pulling decoded frames into JavaScript memory (which forces expensive CPU-GPU data copies), WebGPU lets you import a VideoFrame directly as a GPUExternalTexture. From there, the entire pipeline — effects, transforms, analysis — stays on the GPU.

This enables:
- Real-time color effects (hue shift, brightness, contrast, saturation) running as WGSL fragment shaders
- Heavy image effects (pixelate, kaleidoscope, RGB split, box blur) without CPU involvement
- GPU compute for analysis: histogram, waveform, vectorscope, and optical flow
- Ping-pong texture patterns for chaining multiple effects without readback

The optical flow implementation is particularly impressive — it runs grayscale conversion, pyramid downsampling, spatial/temporal gradients, and Lucas-Kanade flow entirely in compute shaders. The CPU only receives compact statistics (motion totals, scene-cut hints), not full frames.

The project demonstrates that "JavaScript orchestrates a GPU pipeline" is fundamentally different from "JavaScript edits video." The first approach scales.
